# Ogwu — Future Implementation Roadmap

All items below are unimplemented. Sorted by priority (5 = highest).

| Feature | Description | Implementation Method | Priority |
|---|---|---|:---:|
| **LangGraph — Agentic Orchestration** | Replace the flat Vercel AI SDK tool-call loop with LangGraph's stateful directed graphs. Nodes per step (triage, booking, escalation), conditional edges driven by urgency, built-in Postgres checkpointing for fault tolerance, and native human-in-the-loop interrupt before booking confirmation. Currently a failed mid-chain tool call breaks the entire flow with no recovery. | Install `@langchain/langgraph`; define state schema `{ messages, urgency, patient_id, booking_state, tool_results }`; port each agent tool to a LangGraph node; add conditional edge `urgency = emergency → escalate_node`; configure Supabase Postgres checkpointer; replace `agent.js` linear chain; wire LangGraph streaming output to existing SSE response format | 5 |
| **RAG — AWS Comprehend Medical** | Extract named medical entities (ICD-10-CM, RxNorm, SNOMED CT) from patient free-text before passing to the LLM. Grounds urgency classification in structured medical codes rather than unguided model inference. Not for diagnosis — improves triage question relevance and urgency scoring accuracy. | Add `@aws-sdk/client-comprehendmedical` to backend; call `InferICD10CM` on each triage answer; filter entities above 0.80 confidence score; inject structured entity payload into LangGraph triage node state; update `computeUrgency` to weight high-confidence emergency-coded entities more heavily | 5 |
| **RAG — AWS HealthLake (FHIR R4 store)** | Replace raw Supabase JSON blobs for clinical data with a FHIR R4-compliant, queryable, versioned health data layer. At triage time, prior conditions and medications are injected into agent context automatically — no re-asking information the system already has. | Provision HealthLake datastore (us-east-1); configure SMART on FHIR + OAuth 2.0; map `triage_intakes` answers → FHIR Observation resources on save; map `encounters` → FHIR Encounter resources; query patient's prior Condition and Medication resources at triage entry and inject into LangGraph state | 5 |
| **Evaluation & Guardrails** | Currently missing entirely. Urgency classifications are not validated before storing; agent responses have no content safety check; emergency triage results only return a badge with no human notification; no automated evals exist to catch prompt regressions. This is the highest safety risk for a health app. | Add Zod enum guard on urgency output (`routine \| soon \| urgent \| emergency`) before DB write; add OpenAI moderation API call before sending agent responses to patients; configure LangGraph interrupt node on `urgency = emergency` to notify a provider immediately; build eval fixture suite (known symptom inputs → expected urgency); add schema validation on all LLM-generated JSON before use | 4 |
| **Push Notifications** | Patients have no signal when a provider replies to their consult thread. Without this, the async consult loop is broken in practice — patients must manually poll. | Add `expo-notifications` to mobile; store Expo push token in `profiles` table on login; backend sends push via Expo Push API when a `messages` row is inserted by a provider role; handle token expiry and re-registration | 4 |
| **Quick Reply Chips — Triage** | 2–3 contextual short-answer chips appear per triage question (similar to Google Chat smart replies). Patients tap a chip to populate the answer field instead of typing, reducing friction especially on mobile. Free-text input remains available alongside chips. | Add `suggestionsForQuestion(q)` to `AppRouter.tsx` — keyword-matches question text (duration, severity, allergies, fever, medications, conditions) to return 2–3 answers; render as horizontal `ScrollView` of pill buttons in `TriageScreen.tsx` above the text input; selected chip highlights purple; coexists with free-text | 4 |
| **Voice Input — Triage** | Mic button on the triage screen lets patients speak answers instead of typing. Audio is recorded on-device, sent to the backend, and transcribed via OpenAI Whisper before populating the answer field. | Install `expo-av` + `expo-file-system`; add mic/stop toggle button in `TriageScreen.tsx`; on stop, read recording as base64 via FileSystem; `POST /api/triage/transcribe`; backend converts base64 → Blob → FormData → OpenAI Whisper API (`whisper-1`); transcribed text populates answer field; add `NSMicrophoneUsageDescription` to `app.json` | 4 |
| **Bedrock Data Automation — Patient Records** | Patients upload personal health records (PDFs, discharge summaries, lab results). Bedrock Data Automation (BDA) extracts structured data (conditions, medications, dates, provider names); output flows into HealthLake as FHIR resources and enriches the patient's triage context. | S3 bucket with `patients/` prefix; EventBridge ObjectCreated rule (filter `.pdf`/`.docx`) → extraction Lambda (invoke BDA async); BDA output → chunk + embed → SQS → embedding Lambda → pgvector; DLQ + CloudWatch DLQ-depth alarm; mobile document upload screen with file picker | 4 |
| **Bedrock Data Automation — Admin Documents** | Hospital admins upload policy documents, formularies, and care protocols. BDA extracts and chunks content; embeddings stored in pgvector to power the regulatory assistant chat. Hospital knowledge stays current without code changes — re-upload triggers re-embedding. | Reuse S3/EventBridge/Lambda pipeline with `hospitals/` prefix; BDA document blueprint tuned for policy/formulary content; chunk embeddings stored in `hospital_document_chunks` (pgvector); triggers Bedrock Knowledge Base sync; document library UI in admin dashboard with upload, re-process, and delete | 4 |
| **Tool Use Hardening** | Agent tools currently lack strict schema definitions and failure contracts. A tool returning unexpected output silently corrupts the agent's reasoning. Booking tool has read access to data it shouldn't need. No tool call logging exists for observability. | Add Zod schemas for all tool input/output in `agent.js`; add per-tool error envelope `{ ok, data, error }` so the agent can reason about failures; scope tool permissions (booking tool cannot access drug-interaction data); log each tool call + result to a `tool_call_logs` table in Supabase | 4 |
| **Feedback Loops & Error Recovery** | Agent currently fires tool calls and moves on regardless of result. No retry logic, no fallback paths when booking fails, and no escalation to a human after repeated failures. Errors surface as generic "something went wrong" messages. | Check tool response `ok` status in each LangGraph node before proceeding; exponential backoff (3 retries, 1s/2s/4s) on network and rate-limit errors; fallback edge: booking failure → offer alternative times or different hospital; escalation node: after 3 failures, route to provider inbox with full agent context attached | 4 |
| **Hospital Knowledge Base (RAG Pipeline)** | Hospital-specific vector store built from uploaded documents. The triage and booking agent retrieves relevant policy and formulary chunks before generating responses, enabling hospital-specific reasoning (intake policies, covered procedures, after-hours protocols) without hardcoding. | Enable pgvector in Supabase; create `hospital_documents` + `hospital_document_chunks` tables with RLS (hospital sees only its own chunks); chunking utility (500-token segments, 50-token overlap); `POST /api/documents` ingest endpoint; `POST /api/documents/query` retrieval endpoint; inject top-5 chunks (cosine similarity > 0.75, capped at ~2000 tokens) into agent system prompt | 3 |
| **Regulatory Assistant — Admin Dashboard** | New `/knowledge` page in the web admin where hospital admins query their uploaded documents in natural language. Each answer cites the source document and section. Replaces manual document review for compliance and policy questions. | New `/knowledge` React route; chat UI reusing existing `chat-input-bar` / `chat-bubble` CSS classes; queries hit `POST /api/documents/query`; response includes `[Document: {name}, Section {n}]` citation rendering; add Knowledge Base entry to sidebar nav; suggested starter questions pre-populated | 3 |
| **Kafka (AWS MSK) — Appointment Events** | Replace synchronous booking flow with event-driven architecture. Decouples booking confirmation from downstream actions (push notifications, HealthLake updates, calendar sync, analytics) so a failure in one consumer does not block or roll back the booking. | Provision MSK Express cluster (single-AZ dev, multi-AZ prod); define topics: `ogwu.appointment.requested`, `ogwu.appointment.confirmed`, `ogwu.appointment.cancelled`, `ogwu.document.uploaded`; refactor `agent.js` booking tool to publish to Kafka instead of direct DB write; separate Lambda consumers per downstream action; Avro schema + Schema Registry; dead-letter topics + consumer-lag CloudWatch alarm. Note: revisit after launch — SQS is sufficient below ~500 bookings/day | 3 |
| **Auth Token Refresh** | Supabase session expiry mid-operation silently breaks requests on both mobile and web admin. No refresh interceptor exists — users get cryptic auth errors instead of a seamless re-auth. | `supabase.auth.onAuthStateChange` listener in `AppRouter.tsx` to detect token expiry; token-refresh interceptor in `lib/api.ts` to retry requests after refresh; clear sensitive state (`profile`, triage data, thread data) on sign-out in web `App.tsx` | 3 |
| **Pagination — List Endpoints** | Consults, patients, and threads fetch all records at once. Will cause slow loads and out-of-memory issues under real patient volume. | Add `limit` / `offset` (or cursor-based) query params to `/api/threads`, `/api/encounters`, `/api/patients`; update frontend list components with `FlatList` `onEndReached` infinite scroll; backend returns `{ data, next_cursor, has_more }` | 3 |
| **Structured Logging** | `console.log` / `console.error` provide no queryable audit trail in production. No request duration, user context, or structured error fields exist. | Add `pino` to backend; emit `{ level, timestamp, route, user_id, duration_ms, error }` per request; replace all `console.log` / `console.error` calls; ship logs to CloudWatch Logs or Datadog | 3 |
| **Automated Test Suite** | Currently zero tests. Backend auth routes and data mutation paths are the highest risk for silent regressions, especially as the triage prompt and agent tools evolve. | Add `jest` + `supertest` to backend; write integration tests for triage, auth, booking, and thread routes against a dedicated Supabase test project; add Expo test runner (`jest-expo`) for critical mobile flows; run on CI (GitHub Actions) on every PR | 3 |
| **Context & Memory Management** | Conversation history grows unbounded and is naively re-sent every request. Near the model's token limit, patient context is silently truncated — the agent loses critical prior answers. | Implement sliding-window trim in `buildSystemPrompt.js` (drop oldest messages first, preserve system context and triage answers); enforce token budget check before each LLM call; log prompt token count per request to monitor growth over sessions | 3 |
| **Agent Planning & Reasoning** | System prompt does not instruct the agent to plan before acting. No reasoning trace is logged — when the agent produces a wrong urgency or bad booking, there is no way to diagnose why. | Add chain-of-thought instruction to system prompt ("think step by step before calling any tool"); add `done` decision node in LangGraph graph (agent chooses to stop asking vs. continue); log full reasoning trace per agent run to `agent_runs` table, not just the final output | 3 |
| **React Error Boundaries** | Unhandled React errors in the web admin crash the entire UI with a blank screen and no recovery path for the user. | Wrap top-level admin routes in `<ErrorBoundary>` components; add fallback UI (error message + reload button); forward error + component stack to Sentry on catch | 2 |
| **Graceful Backend Shutdown** | Railway redeploys kill the Node process immediately with no drain. In-flight requests (triage LLM calls, booking writes) are dropped mid-execution. | Handle `SIGTERM` in `src/index.js`; stop accepting new connections; `await` in-flight requests to complete with a 10-second hard timeout; then `process.exit(0)` | 2 |
| **Dockerfile** | Backend relies on Railway's auto-detection with no reproducible build artifact. Local dev and production environments can silently diverge. | Write multi-stage `Dockerfile` (build stage + slim runtime stage); add `.dockerignore`; expose `PORT` env var; add to Railway deployment config; test build locally before merging | 2 |
| **EAS Build Profiles** | Mobile app has no staging/production distinction. All builds point to the same backend URL and Supabase project. | Add `development`, `staging`, `production` profiles to `eas.json`; per-profile bundle IDs and `EXPO_PUBLIC_API_URL` / Supabase key overrides; configure `app.json` `extra` field for per-environment values | 2 |
| **Helmet Middleware** | Backend is missing HTTP security headers: no CSP, no HSTS, no X-Frame-Options, no referrer policy. | Add `helmet` to `src/index.js` middleware chain; configure `contentSecurityPolicy` to allow only known origins and CDNs; enable HSTS in production only | 2 |
| **Sentry / Error Tracking** | No runtime error alerting in production. Errors are only visible if someone reads Railway logs. Silent crashes and slow transactions go undetected. | Add `@sentry/node` to backend; `@sentry/react-native` to mobile; configure DSN via env var; capture unhandled exceptions, slow DB queries, and failed tool calls; set up alert rules for error rate spikes | 2 |
| **Read Receipts & Typing Indicators** | Consult thread UI has no real-time feedback. Patients and providers cannot tell if the other party is actively engaged or has read their message. | Use Supabase Realtime channels on the `messages` table; emit typing indicator via Realtime presence; mark messages as read on view and persist `read_at` timestamp; update thread list with unread count badge | 2 |
| **Password Policy** | Supabase Auth default requires only `length >= 8`. No complexity enforcement exists. | Enforce complexity rules (uppercase, number, symbol) via Supabase Auth dashboard password policy settings, or via a pre-signup Edge Function that rejects weak passwords before account creation | 2 |
| **Infinite Scroll** | Patient and consult list pages load all records in a single query. Page load time degrades linearly with record count. | Replace flat list renders with `FlatList` + `onEndReached` callback; backend returns cursor-based pages `{ data, next_cursor }`; add bottom loading spinner while next page fetches | 2 |
| **`.env.example` Files** | No documentation of required environment variables. New developers and new deployments have no reference for what keys are needed or what format they take. | Add `.env.example` to `backend/`, `mobile/`, and web admin with all required keys listed, short descriptions, and safe dummy values; reference in each package README | 2 |
